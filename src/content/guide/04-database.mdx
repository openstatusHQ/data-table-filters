# Database

## Drizzle ORM

A set of helper functions to implement the server-side API with [Drizzle ORM](https://orm.drizzle.team) and PostgreSQL. See the [live example](/d).

### DB Schema

Define your PostgreSQL table with Drizzle. See [`src/db/drizzle/schema.ts`](https://github.com/openstatusHQ/data-table-filters/blob/main/src/db/drizzle/schema.ts) for the full example schema (enums, columns, types).

### Column Mapping

The bridge between your table schema field keys and Drizzle table columns. Supports dotted names for nested fields (e.g. `"timing.dns"` maps to a flat `timingDns` column). One file per table:

```ts
import { logs } from "@/db/drizzle/schema";
import type { ColumnMapping } from "@/lib/drizzle";

export const columnMapping = {
  level: logs.level,
  date: logs.date,
  status: logs.status,
  latency: logs.latency,
  host: logs.host,
  regions: logs.regions,
  "timing.dns": logs.timingDns,
} satisfies ColumnMapping;
```

### Handler

The easiest way to set up the server side is with `createDrizzleHandler`, which wraps all helper functions into a single call:

```ts
import { createDrizzleHandler } from "@/lib/drizzle";

const handler = createDrizzleHandler({
  db,
  table: logs,
  schema: tableSchema.definition,
  columnMapping,
  cursorColumn: "date",
  defaultSize: 40,
});

// In your API route
const result = await handler.execute(search);
// → { data, totalRowCount, filterRowCount, facets, prevCursor, nextCursor }
```

### Helper Functions

If you need more control, four lower-level functions cover the full query lifecycle:

```ts
import {
  buildWhereConditions,
  buildOrderBy,
  buildCursorPagination,
  computeFacets,
} from "@/lib/drizzle";
```

- `buildWhereConditions(mapping, filters, options?)` — converts filter state to SQL WHERE clauses. Handles `ilike` for strings, `between` for slider ranges, `inArray` for checkboxes, date ranges, and PostgreSQL array overlap (`&&`) for array columns.
- `buildOrderBy(mapping, sort)` — returns an `asc` or `desc` SQL clause from a sort descriptor.
- `buildCursorPagination({ cursor, direction, size, cursorColumn })` — returns cursor condition, order clause, and a `needsReverse` flag for infinite scroll.
- `computeFacets(db, table, mapping, conditions, facetKeys, options?)` — computes grouped counts (or min/max for slider keys) via parallel SQL queries.

### Three-Pass Filtering

The API route uses a three-pass strategy so slider bounds don't collapse when you adjust them:

1. **Date only** — base time range, used for chart data
2. **Date + non-slider filters** — used to compute slider min/max (so moving a slider doesn't shrink its own bounds)
3. **All filters** — used for the final data query and non-slider facets

The key insight: slider facets need to account for date and checkbox filters but must ignore _other_ sliders. Otherwise, adjusting one slider would narrow the range of another. The three-pass approach keeps slider bounds stable while still reflecting active filters.

The `sliderKeys`, `facetKeys`, and `dateKeys` are automatically derived from your table schema definition based on each column's filter type.
